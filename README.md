Учебный сервис потоковой трансляции биржевых котировок на Rust.

Клиент подписывается на набор тикеров через TCP; сервер затем отправляет симулированные котировки обратно по UDP. Клиент периодически шлёт PING-датаграммы, сигнализируя что он ещё жив — если пинги прекращаются, сервер закрывает поток.

```
Клиент ──── TCP-рукопожатие ────► Сервер :7878
       ◄─── UDP-котировки ────────
       ──── UDP-PINGи ────────────►
```

---

## Сборка

```bash
cargo build
```

Запуск тестов:

```bash
cargo test
```

> Сервер читает `assets/all_tickers.txt` при старте **из текущей рабочей директории**, поэтому всегда запускайте его из корня репозитория.

---

## Запуск

### 1. Запуск сервера

```bash
cargo run --bin server
```

Сервер слушает `127.0.0.1:7878` (TCP). При успешном старте вы увидите:

```
INFO server: TCP server listening on 127.0.0.1:7878
```

### 2. Запуск клиента

```bash
cargo run --bin client -- \
    --server-addr 127.0.0.1:7878 \
    --udp-port 9000 \
    --tickers-file assets/all_tickers.txt
```

| Флаг             | Значение по умолчанию | Описание                                           |
| ---------------- | --------------------- | -------------------------------------------------- |
| `--server-addr`  | `127.0.0.1:7878`      | TCP-адрес сервера                                  |
| `--udp-port`     | _(обязательный)_      | Локальный UDP-порт, на котором слушает клиент      |
| `--tickers-file` | _(обязательный)_      | Путь к файлу тикеров (по одному символу на строку) |

Можно запускать несколько клиентов одновременно на разных UDP-портах с разными файлами тикеров:

```bash
# Терминал 2 — подписка на все тикеры из спека
cargo run --bin client -- --udp-port 9001 --tickers-file assets/all_tickers.txt

# Терминал 3 — только AAPL
cargo run --bin client -- --udp-port 9002 --tickers-file assets/appl.txt
```

## Формат файла тикеров

Обычный текстовый файл, по одному тикеру на строку. Пустые строки пропускаются, пробелы по краям обрезаются. Символы сопоставляются без учёта регистра.

```
AAPL
MSFT
GOOGL
AMZN
TSLA
```

Правила:

- Один тикер на строку.
- Пустые строки пропускаются.
- Пробелы по краям обрезаются (`trim`).
- Символы проверяются по авторитетному списку сервера (`assets/tickers.txt`). Любой неизвестный символ приводит к отклонению всего рукопожатия.

Поддерживаемые тикеры (на стороне сервера): `AAPL`, `MSFT`, `GOOGL`, `AMZN`, `TSLA`, `NVDA`, `META`, `NFLX`, `AMD`, `INTC`.

## Проводной протокол

### Рукопожатие (TCP)

Клиент открывает TCP-соединение к порту 7878 и отправляет одну строку, оканчивающуюся на `\n`:

```
STREAM <ip:udp-порт> <ТИКЕР1,ТИКЕР2,...>\n
```

Пример:

```
STREAM 127.0.0.1:9000 AAPL,MSFT,NVDA\n
```

Сервер отвечает одной строкой, после чего закрывает соединение:

| Ответ             | Значение                               |
| ----------------- | -------------------------------------- |
| `OK\n`            | Подписка принята; UDP-поток начнётся   |
| `ERR <причина>\n` | Отклонено — причина описывает проблему |

### Котировочные датаграммы (UDP)

Каждая датаграмма содержит ровно одну котировку — одну строку, поля разделены символом `|`, пробелы внутри полей не допускаются:

```
ТИКЕР|ЦЕНА|ОБЪЁМ|ВРЕМЕННАЯ_МЕТКА_МС
```

Пример:

```
AAPL|189.45|1000000|1718000000000
```

| Поле                 | Тип    | Описание                                           |
| -------------------- | ------ | -------------------------------------------------- |
| `ТИКЕР`              | string | Символ тикера, заглавные буквы                     |
| `ЦЕНА`               | f64    | Симулированная цена в USD, два знака после запятой |
| `ОБЪЁМ`              | u32    | Симулированный объём сделки                        |
| `ВРЕМЕННАЯ_МЕТКА_МС` | u64    | Unix-время в миллисекундах                         |

Ограничения формата:

- Пробелы внутри полей не допускаются.
- Одна котировка — одна UDP-датаграмма.
- Буфер приёма: 2 КиБ достаточно (каждая строка значительно короче 100 байт).

Котировки генерируются каждые **300 мс**.

## Проверка фильтрации тикеров

Запустите двух клиентов с разными наборами тикеров и следите за их логами:

```bash
# Клиент A — только AAPL
echo "AAPL" > /tmp/one.txt
cargo run --bin client -- --udp-port 9001 --tickers-file /assets/aapl.txt

# Клиент B — AAPL и MSFT
echo "MSFT" > /tmp/two.txt
cargo run --bin client -- --udp-port 9002 --tickers-file /assets/aapl_msft.txt
```

В логах клиента A должны быть только строки `[AAPL]`, в логах клиента B — только `[MSFT]`. Это подтверждает работу фильтрации по подписчикам на стороне сервера.

## Проверка таймаута PING

Клиент отправляет датаграмму `PING` на UDP-сокет сервера каждые **2 секунды**. Если сервер не получает ни одного PING в течение **5 секунд** — он закрывает UDP-отправитель для этого подписчика.

Чтобы воспроизвести таймаут, завершите клиент (`Ctrl-C`) и следите за логом сервера:

```
INFO  server: client 127.0.0.1:9000 timed out, closing UDP sender
INFO  server: UDP sender for 127.0.0.1:9000 exiting
```

Константы таймаута и интервала находятся в `src/lib.rs` (`PING_TIMEOUT`, `PING_INTERVAL`) — их можно уменьшить перед сборкой, чтобы ускорить наблюдение эффекта.

## Известные ограничения

- **Только localhost** — сервер привязывается к `127.0.0.1`; удалённые клиенты не поддерживаются.
- **Только IPv4** — адреса IPv6 не разбираются и не принимаются протоколом.
- **Одна копия сервера** — кластеризация не поддерживается; все подписчики разделяют один внутрипроцессный генератор котировок.
- **Нет переподключения** — при перезапуске сервера клиентов также нужно перезапустить вручную.
- **Статичный список тикеров** — `assets/all_tickers.txt` читается один раз при старте; изменения требуют перезапуска сервера.
